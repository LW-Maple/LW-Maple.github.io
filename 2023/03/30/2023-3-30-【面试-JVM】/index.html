JVM包含哪几部分?

  • ClassLoader（类加载器）
    • 负责加载字节码文件即 class 文件
  • Runtime Data Area（运行时数据区）
    • Heap（堆），Method Area（方法区）
    • Native Method Stack（本地方法栈）,Stack（虚拟机栈）,PC Register（程序计数器），
    • 几乎所有的关于 Java 内存方面的问题，都是集中在这块
  • Execution Engine（执行引擎）
    • 负责把命令解释给操作系统，即将 JVM 指令集翻译为操作系统指令集
  • Native Interface（本地库接口）

运行时数据区包含哪几部分?

  • 首先,根据是否线程共享分为两大区域
    • 线程共享
      • 堆
        • 几乎所有的对象都存放在堆
        • 因此也是GC的主要区域
      • 方法区
        • 主要是存放一些共享的信息
        • 类,常量,静态变量
    • 线程私有的
      • 虚拟机栈
        • 用于管理java方法的执行
        • 栈内存储的叫做栈帧,对应的就是一次方法的调用
        • 调用入栈,方法完成后出栈
          • 每一个栈帧又含有,本地变量表啊
          • 返回的值啊,等等等
      • 本地方法栈,管理本地方法
      • 程序计数器
        • 管理程序的执行,见下方问题

java程序是如何运行的?

  • 首先是我们写好的程序,经过编译器的编译

    • 词法,语法,语义分析,最后生成字节码文件

  • 类加载器,把字节码文件加载到运行时数据区(包含的是类的信息)

  • 之后,通过执行引擎将字节码翻译为机器码,通过操作系统操作 CPU 执行获取结果

类加载的过程?

加载->验证->准备->解析->初始化

  • 加载
    • 获取类的二进制流
    • 转换为运行时数据区的数据结构
    • 生成这个类的Class对象
  • 验证
    • 确保Class文件符合规范,不会危害JVM
  • 准备
    • 为类的静态变量初始化(初始零值)
  • 解析
    • 符号引用转为直接引用
    • 举个例子 sout 就是符号引用,直接引用,就是调用这个方法的具体内存地址
  • 初始化
    • 调用类构造器<cinit>,也是进行初始化,根据程序员通过程序编码制定的主观计划去初始化类变量

对象实例化的过程?

1.查询对象所属的类是否进行了初始化,也就是去找类的元信息,没有的话会报ClassNotFound异常

2.计算对象的内存大小,为其分配内存空间

3.对象初始化,分三步, 默认初始化,显示初始化

4.设置对象的对象头,哈希码,锁啊,GC相关的年龄啊等等

5.最后才是构造器初始化

双亲委派模型?

加载类需要用到类加载器,加载器在java中分三层

  • 启动类加载器,它用来加载Java的核心库
  • 扩展类加载器,扩展类
  • 应用程序类加载器它负责加载（ClassPath）上所有的类库

简单点说就是一但类加载器收到了加载类的委托,它并不会立即加载,而是交个父类去加载,直到父类无法加载该任务,子类加载器才会去执行加载类的任务

可以保证核心API的安全

如何打破双亲委派？

自己写个类继承classLoader重写loadclass方法

然后了解的像是JDBC里加载mysql驱动

本地方法栈有什么用？

管理本地方法的运行

程序计数器?

这个概念其实来源于计组,程序计数器,也叫PC寄存器

是用于存储下一跳指令地址的寄存器,是程序能够实现顺序、分支、循环的核心

JAVA中的计数器是对物理PC寄存器的虚拟实现,目的也是控制程序的执行

逃逸分析?

当一个对象在方法里面被定义后，它可能被外部方法所引用，例如作为调用参数传递到其他方法中，这种称为方法逃逸；

堆?

是大多数实例对象存在的位置

具体细分为

新生区,养老区,和元空间

  • 新生区:养老区默认是1:2

新生区分为

伊甸园区:是对象刚创建后存放的位置(此区域满时会触发YGC)

幸存者1

幸存者0

这俩要看成整体,因为区分年代是通过年龄计数的

每熬过一次GC就回+1年龄

YGC后的对象会在0,1之间反复移动,并进行年龄计数

然后在这样反复的年龄计数,默认是15, 就会转去养老区

养老区满了也GC,清除不了就回OOM

至于为啥要分俩个区域,是为了防止内存碎片

为何要进行以分代的思想进行划分呢?

主要是为了服务垃圾回收

建立在两个假说上,绝大多数对象都是朝生夕灭的,熬过越多次垃圾收集过程的对象就越难以消亡

所以垃圾回收的方向,就转向为了如何去保存那些难以消灭的对象

将它们放在一起,这篇区域可以使用较低的频率来回收这个区域，这就同时兼顾了垃圾收集的时间开销和内存的空间有效利用

垃圾回收过程?

什么是垃圾?

垃圾是指在运行程序中没有任何指针指向的对象，这个对象就是需要被回收的垃圾。

垃圾回收的区域

  • 堆 这是主要的区域,因为大多数实例化的对象都在这里
  • 非堆区,(废弃的常量和不再使用的类型)

垃圾标记阶段算法

可达性分析算法

定义一组根对象,从跟对象出发搜索被根对象集合所连接的目标对象是否可达。

扩展:引用计数算法

根对象可以是哪些?

各个线程被调用的方法中使用到的参数、局部变量等。

Java类的引用类型静态变量

垃圾回收算法

标记清除算法

标记:从根对象出发,标记所有可达的对象(注意是可达的)

从堆内存线性遍历,然后清除所有未可达的对象

  • 缺点是会造成程序停顿
  • 会产生大量内存碎片

复制算法

将内存分为两块 每次只用其中一块内存,这样可以保证必定有一块是空内存

将正在使用的对象放入另一块空内存区域,然后清除正在使用的内存区域,交换两个内存区域的角色

  • 优点是不会存在内存碎片
  • 缺点是需要两倍的内存区域

长用在新生代

老年代都为长存活对象,多次复制影响效率

标记压缩算法

  • 第一步先标记可达对象
  • 第二步将其压缩到内存的一端,按顺序存放
  • 第三步清除剩余的内存

优点是没有内存碎片,没有额外的内存开销

缺点是效率降低,移动过程中会暂停用户程序

垃圾回收器?

G1

把java堆划分为多个大小相等的独立区域（Region），每一个Region都可以根据需要，扮演新生代的Eden空间、Survivor空间，或者老年代空间

有什么好处呢?

  • 它将Region作为单次回收的最小单元，即每次收集到的内存空间都是Region大小的整数倍，这样可以有计划地避免在整个Java堆中进行全区域的垃圾收集
  • 可以去计算每个单元的垃圾价值,回收最高价值的垃圾

保证了G1收集器在有限的时间内获取尽可能高的收集效率。

CMS

了解较少,只知道是低延迟的垃圾回收器,多个线程工图清理,减少等待时间

完整的GC流程?

当最后一个对象放入伊甸园放不下的时候了,触发minorGC

尝试放入幸存者区,放的下,皆大欢喜

放不下,尝试放入老年代, 放得下皆大欢喜

放不下,进行fullGC, 还放不下 报OOM

FULL GC是什么?

清理所有堆内存中的垃圾

Full GC会“Stop The World”，即在GC期间全程暂停用户的应用程序。

内存溢出与内存泄漏?

  • 内存溢出

是指程序在申请内存时，没有足够的内存空间供其使用

修改JVM启动参数，直接增加内存

  • 内存泄漏

是指程序在申请内存后，无法释放已申请的内存空间。

  1. 尽早释放无用对象的引用。

哪些地方会OOM

堆,不断创建对象,直至堆满

方法区

栈

超出固定栈大小，则抛出StackOverFlowError错误，比如进行了一个不会停止的递归调用；
栈可以动态拓展的，拓展时无法申请到足够的内存，则抛出OutOfMemoryError错误。

直接内存

Unsafe 申请

对象内存布局?

对象头占 12B，包括对象标记和类型指针。对象标记存储对象自身的运行时数据，如哈希码、GC 分代年龄、锁标志、

实例数据是对象真正存储的有效信息，即本类对象的实例成员变量和所有可见的父类成员变量。。

对齐填充不是必然存在的，仅起占位符作用。

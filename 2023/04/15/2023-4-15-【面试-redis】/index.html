面试-redis

是C语言开发的一个开源高性能键值对的内存数据库

redis为什么快?

1.Redis 基于内存，内存的访问速度是磁盘的上千倍；

2.采用了高效的数据结构，

3.使用单线程可以省去多线程时CPU上下文会切换的时间，也不用去考虑各种锁的问题，不存在加锁释放锁操作，没有死锁问题导致的性能消耗。

为什么要用 Redis/为什么要用缓存？

为了高性能

  • 如果说，用户访问的数据属于高频数据并且不会经常改变的话，那么我们就可以很放心地将该用户访问的数据存在缓存中。

一般像 MySQL 这类的数据库的 QPS 大概都在 1w 左右（4 核 8g） ，但是使用 Redis 缓存之后很容易达到 10w+，甚至最高能达到 30w+

  • 提高系统能够承受的请求数量

缓存更新策略

旁路缓存模式

先更新数据库,后删缓存

  • 适合读多写少

分布式锁

分布式锁工具基于 Redisson 来实现分布式锁。

数据结构

  • 5 种基础数据结构 ：String（字符串）、List（列表）、Set（集合）、Hash（散列）、Zset（有序集合）。

  • 3 种特殊数据结构 ：HyperLogLogs（基数统计）、Bitmap （位存储）、Geospatial (地理位置)。

    • 位图,以字为单位的数组,只能填0,1
    • geo,地理位置的数据结构,提供了经纬度的相关支持
    • hyperloglog 基数统计
      • 基数:数据集中不重复的元素个数
      • 传统计数会占用大量的存储空间
      • 牺牲小部分准确度的情况下进行计数
      • 基于算法实现

  • string

常规数据,token、序列化后的对象、图片的路径

分布式锁(利用 SETNX key value 命令可以实现一个最简易的分布式锁)

  • sortedset

排行榜

  • set

点赞功能

求集合运算,交并补

  • bitmap

签到

  • hyperloglog

UV统计

持久化

RDB

数据定时写入磁盘,恢复时读入内存

AOF

保存日志,(存操作,不存数据),恢复加载写操作

生产问题

缓存穿透:

大量请求的 key 是不合理的，根本不存在于缓存中，也不存在于数据库中 。这就导致这些请求直接到了数据库上，根本没有经过缓存这一层，对数据库造成了巨大的压力

  • 缓存空值,白名单

缓存击穿:

请求的 key 对应的是 热点数据 ，由于该数据过期,导致大量请求到了数据库上

  • 逻辑过期,提前预热,加锁

缓存雪崩:

缓存在同一时间大面积的失效，导致大量的请求都直接落到了数据库上，对数据库造成了巨大的压力.

  • 分散过期时间
